# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Python toolbox (Windows-oriented) for analysing Mirth Connect log files, monitoring system resources, and emailing the resulting reports. Four standalone scripts at the repo root (`mirth_logs_parser.py`, `system_state.py`, `quickmail.py`, `mirth_api.py` — the latter a Mirth Connect REST client) run directly via CLI or through the interactive `_cmd_helper/launch.bat` menu. On top of them, **`checker_service.py`** is a long-running web service (the `lib/` package + `web/` pages) that reuses these scripts as libraries, records a time-series in SQLite, and serves dashboards.

## Environment & common commands

The `.bat` helpers in `_cmd_helper/` all `cd` to the repo root first and (except `venv_create`) auto-create the venv if missing. They are written for `cmd.exe`; this machine's default shell is PowerShell, so call them as `cmd /c _cmd_helper\launch.bat` if needed.

- Create venv: `_cmd_helper\venv_create.bat` (creates `venv/` at root)
- Load venv interactively: `_cmd_helper\venv_load.bat` (pass `--no-shell` to set up the env without spawning a sub-shell)
- Install/update deps: `_cmd_helper\update.bat` (upgrades pip, installs `requirements.txt`)
- Interactive launcher: `_cmd_helper\launch.bat` — lists root `.py` files (skips dotfiles like `.smtp_config.py`), shows each script's `-h` syntax, then prompts for args
- Build standalone `.exe`s: `_cmd_helper\_compilation.bat` (PyInstaller `--onefile` → `dist/`)

Running scripts directly:
- `python mirth_logs_parser.py [logfile] [-d DATE] [-r] [-m EMAIL]` — `-d 1`=all, `0`=today, `-1`=J-1, `-X`=J-X; `-r` also parses logrotate files (`{filename}.x`); `-m` emails the report. Default logfile is the standard Mirth Connect server log `C:\Program Files\Mirth Connect\logs\mirthconnect.log` (the bundled `Ressources\mirth-exemple.log` remains available as a sample).
- `python system_state.py [-c "chrome,python"]` — `-c` filters to named processes (comma-separated)
- `python quickmail.py -s SUBJECT -m MESSAGE -d DEST` — direct send; exits 0 on success, 1 on failure
- `python mirth_api.py [-t TIMEOUT] [-s server,channels,connectors,stats,errors] [-c NAME] [-f/--full] [-j/--json] [-l LIMIT]` — queries the Mirth REST server and prints a tabulate report (`connectors` = per-connector stats; `--full` adds the errored-messages detail; `--json` dumps the full report); exits 1 if the server is unreachable. Config from env / `.mirth_config.py` (see Web service section).

There is no test suite, linter, or CI configured.

## Architecture

**`mirth_logs_parser.py`** — the data layer is three pure functions (`mirth_file_parser` groups multi-line entries; `mirth_log_decoder` extracts level/channel/thread/class/`Caused by:`; `mirth_log_filter`). Everything CLI- and presentation-related (`display_statistics`, table printing, HTML report generation, `main()`) lives *inside* the `if __name__ == '__main__':` block, so importing the module gives you only the parsing primitives. When `-m` is set, stdout from `display_statistics` is captured via `contextlib.redirect_stdout` into a buffer, then converted to HTML (`format_report_to_html`) and sent through `quickmail.sendmail`.

**`system_state.py`** — a flat library of `get_*` probe functions (hostname, CPU, memory, disk, network I/O, TCP/UDP sockets, VPN detection via Windows adapter descriptions, ping) using `psutil` and `ping3`. The CLI block at the bottom orchestrates these into `tabulate` tables.

**`quickmail.py`** — used both as a CLI and as the import dependency for the parser's email feature. `sendmail()` is the public entry point; it auto-handles SSL vs STARTTLS (STARTTLS is forced when port == 587).

Both `mirth_logs_parser.py` and `system_state.py` define local `safe_print`/`print_table` helpers that fall back from Unicode `fancy_grid` to ASCII tables — needed for Windows terminals that can't encode box-drawing characters. Preserve this fallback when touching output code.

## Web service (`checker_service.py` + `lib/` + `web/`)

Run with `python checker_service.py [--host 0.0.0.0] [--port 8800] [--interval 60] [--stagger 5] [--logfile ...] [--no-browser]`. It needs the venv (`tabulate`, `psutil`). It exposes a JSON API + static dashboards and runs background collectors writing to a SQLite history.

- **`lib/webserver.py`** — minimal `ThreadingHTTPServer` with a `Router` (regex routes, `{param}` segments), static file serving from `web/`, and a `json_transform` hook (the service uses it to ceil-round all floats to 2 decimals). Handlers take a `Request` and return a dict/list (→JSON 200), a `(status, payload)` tuple, or a `Response` (raw bytes/content-type).
- **`lib/scheduler.py`** — `RecurringTask` runs a function on a daemon thread every `interval` s (duration-compensated, interruptible). `start_delay` offsets the first run; **`start_staggered(tasks, step=5.0)`** launches a list of tasks each delayed `n*step` s so their probes don't all hit the machine on the same tick. `main()` runs three collectors (`metrics-collector`, `mirth-collector`, `mirth-overview-collector`) this way. **`mirth-overview-collector`** (`scheduled_mirth_overview`) makes the single per-tick `get_overview()` call and persists the whole snapshot via `database.insert_mirth_snapshot` (replaces the old `mirth-stats-collector`/`mirth_throughput` path).
- **`lib/database.py`** — SQLite (WAL, `PRAGMA foreign_keys=ON`). Tables: **`metrics`** (host system relevés, `tag='system'`), **`mirth_metrics`** (dedicated Mirth-process relevés — CPU/mem/`sockets`; same schema as `metrics`, disk columns unused), **`mirth_messages`** (content cache for errored messages, unique key `(channel_id, message_id, meta_data_id)`), **`events`** (alert markers overlaid on charts), and the **Mirth-overview historisation trio** (replaces the dropped `mirth_throughput` — `init_db` does `DROP TABLE IF EXISTS mirth_throughput`, no migration): **`mirth_entity`** (stable dimension of channels+connectors, unique `(channel_id, meta_data_id)`; `meta_data_id NULL`=the channel, `0`=source, `1+`=destinations), **`mirth_stats`** (per-tick facts — one row per entity, `entity_id` FK → `mirth_entity`, columns `state`+`received/filtered/queued/sent/error`), and **`mirth_server`** (per-tick server snapshot — `reachable`, `version`, `channel_count`, `channels_started`, global totals, `system_stats` JSON, `error_text`). The `metrics`/`mirth_metrics` access functions (`insert_metric`/`get_history`/`get_latest`/`get_last_valid`/`insert_event_marker`) take a `table=` arg whitelisted by `_check_table`; Mirth metrics live in their **own table** (not `metrics(tag='mirth')`). Event markers (`boot`/`restart`) are null-metric rows that break the chart line on outages. Cache helpers: `get_cached_message_keys`/`get_cached_messages(keys)`/`upsert_mirth_messages` (INSERT OR IGNORE). Overview historisation: **`insert_mirth_snapshot(overview)`** (one transaction/tick: upserts entities via `_get_or_create_entity`, inserts one `mirth_stats` row per entity + one `mirth_server` row; if unreachable, writes only the server row with `reachable=0`), **`get_mirth_overview_latest()`** (rebuilds the `get_overview`-shaped dict — channels each with `connectors`, totals, version — from the latest **reachable** `mirth_server` row + its `mirth_stats`, plus `snapshot_at`), **`get_mirth_series(channel_id=None, meta_data_id=None, …)`** (time-series: global ⇒ `mirth_server` totals; channel/connector ⇒ `mirth_stats` JOIN `mirth_entity`). `purge_older_than`/`reset_db` cover all tables (reset deletes facts before the entity dimension, FK-safe).
- **`mirth_api.py`** (repo root, used as a library by the service) — stdlib-only client for the Mirth Connect REST API (cookie-jar login, TLS unverified by default for Mirth's self-signed cert). Same two-part layout as `system_state.py`: importable `get_*` functions that never raise (errors come back as `{reachable: False, error}`) + a `tabulate` CLI (`python mirth_api.py [-t TIMEOUT] [-s server,channels,connectors,stats,errors] [-c NAME] [-f/--full] [-j/--json] [-l/--limit N]`; `--full` adds the errored-messages section, `--json` dumps `build_full_report`). Library entry points: `get_overview()` (full: version + JVM stats + channels **each carrying `connectors`** + `totals`), `get_channels_overview()`, **`get_connectors_overview(channel_id=None)`** (flat connector list), `get_global_statistics()`, `get_server_info()`, **`get_status()`** (minimal live probe: login = reachability + version only — used by `/api/mirth/api` for the real-time état/version while the rest is served from history), `get_errors()`, `get_error_messages(channel_id=None, connector_meta_id=None, limit=50)` (errored messages, optionally filtered to one connector — parsed defensively from `/channels/{id}/messages?status=ERROR&includeContent=true` via `_parse_messages`/`_iter_connector_messages`; each item `{channel_id, channel_name, message_id, connector, meta_data_id, status, received_date, send_attempts (retry), error_code, category, error (full text), content (raw)}`), **`list_error_message_keys()`** (lightweight, `includeContent=false`, no error/content — the authoritative list feeding the cache), **`get_message(channel_id, message_id)`** (single full message), **`build_full_report(include_messages=False)`** (server + channels + connectors + errors). Each channel dict is `{name, channel_id, state, received, filtered, queued, sent, error, connectors:[…]}`; each connector is `{meta_data_id (0=source), name, state, received, filtered, queued, sent, error}` parsed from `childStatuses.dashboardStatus` by `_parse_connectors` (previously only summed). **Statistics parsing is deliberately defensive** (`_parse_statistics`): Mirth serialises the `Map<Status,Long>` with the *enum's fully-qualified class name* as the entry key (`com.mirth.connect.donkey.model.message.Status`), not `string` — so the key is taken as the first non-counter textual value; it also handles plain-dict, `entry`-array, connector aggregation, and an optional `/channels/statistics` fallback. Config precedence mirrors quickmail: env vars (`MIRTH_BASE_URL`, `MIRTH_USER`, `MIRTH_PASSWORD`, `MIRTH_VERIFY_SSL`, `MIRTH_PROCESS`) > `.mirth_config.py` (git-ignored; `.mirth_config.py.template` is the stub) > defaults. `MIRTH_PROCESS` (default `mcservice.exe`) is the process name historised in `mirth_metrics`.

**Error-message cache (`checker_service.cached_error_messages`)** — drives `/api/mirth/messages`. Strategy is **"Mirth is the authority"**: `list_error_message_keys` gives the current errored keys (cheap, no content); only keys absent from `mirth_messages` are downloaded with content (one call per channel that has misses) and cached; the response returns cached content **restricted to the current authoritative key set** (stale cache rows stay in the DB but aren't shown). Optional `connector` (metaDataId) filter. The standalone `mirth_api.get_error_messages` remains the non-cached path (CLI/exe).

Key API routes (full list/playground in `web/api.html`): `/api/history?tag=system|mirth|all&hours=…|date_deb=…&date_fin=…` (`tag=mirth` now reads `mirth_metrics`), `/api/history/latest?tag=…`, `/api/mirth/api` (overview **served from the latest historised snapshot** — channels incl. `connectors` + `totals` + `snapshot_at`; only `reachable`/`version` come from a small live `get_status` call. Falls back to a live `get_overview()` if no snapshot exists yet), `/api/mirth/channels`, **`/api/mirth/connectors?channel=`** (flat connectors), `/api/mirth/stats`, `/api/mirth/server`, `/api/mirth/errors`, `/api/mirth/messages?channel=&connector=&limit=50` (cached; drives the clickable per-channel/per-connector error counters in `web/statistiques.html`), **`/api/mirth/history` + `/api/mirth/history/latest`** (process relevés from `mirth_metrics`), **`/api/mirth/throughput?channel=&connector=&hours=…`** (cumulative counters → client-side msg/min deltas, read via `get_mirth_series`; `channel` omitted ⇒ global totals from `mirth_server`; `channel`+`connector` ⇒ a single connector's series — enables per-channel/per-connector charts), **`/api/mirth/report?messages=0|1`** (`build_full_report`), `/api/getmirthinfo?type=…&format=json|text|html`, `/api/mirth/process` (live process snapshot), `/api/db/*`, `/api/status` (`schedulers` lists every task). Note `tag` absent/empty ⇒ `system`; use `tag=all` for every source.

**Alertes / notifications sortantes (`web/alerte.html` + `alert_methods`/`alert_rules` + `emit_alarm`).** The service turns a fixed catalogue of **known alarms** (`checker_service.ALARM_CATALOG`, in display order: `system_boot`/`checker_down`/`checker_up` → category `systeme`, then `mirth_down`/`mirth_up` → category `mirth` — the same events already drawn as vertical bars) into outbound notifications. Each alarm carries `title`, `event_label` (kept identical to the old `insert_event` label so chart overlays don't change), `category` (drives the event-bar colour via `EVENT_COLORS`; `systeme` added there), `severity` (`critical`/`warning`/`info`), and **per-method defaults** `default_email`/`default_mqtt`. Notification **methods** (`ALERT_METHODS`) are now just `email` (active) + `mqtt` (reserved placeholder, `active:false`; SMS/Slack were removed). Config lives in two *config* tables (untouched by `purge_older_than`/`reset_db`): **`alert_methods`** (one row per method — `enabled`, `recipient`, `config` JSON) and **`alert_rules`** (the OUI/NON matrix, unique `(alarm_code, method)`). DB accessors: `get_alert_config()` (→`{methods, rules}`), `save_alert_config(methods, rules)` (SELECT-then-INSERT/UPDATE, no UPSERT for old-SQLite compat). The 5 emit sites (`scheduled_mirth_check`, `mark_startup_events`) go through **`emit_alarm(code, timestamp=, dedup=)`** — inserts the same event row and, *only when a new row was actually created* (respects `dedup`), spawns a daemon thread running `dispatch_alerts(code, ts)`. **`dispatch_alerts(code, ts, config=None, test=False)` never writes to the DB** (only `emit_alarm` inserts the event) — it builds the context **once** from local DB reads (`_build_alarm_context` — last system/mirth relevés + `get_mirth_overview_latest`, **no network**), then for each enabled method whose method is globally enabled it calls **`_deliver(method, recipient, ctx)`** (email → one `quickmail.sendmail(html=…)` **per recipient**, comma/`;`/newline-split; non-email → a logged "non implémentée" failure result) and returns a list of `{method, dest, ok[, error]}`. `config=` lets a caller impose an unsaved snapshot (used by the per-alarm test). No attachment yet, but `build_alert_email` returns `(subject, message, html)` and the gathered context would make a `.txt` PJ trivial later. Console prints stay cp1252-safe (ASCII `->`, no `→`). Routes (in `build_router`): **`/api/alerts/config`** (GET — catalogue + methods + stored config), **`/api/alerts/save`** (POST — filtered against `ALARM_BY_CODE`/`ALERT_METHODS`), **`/api/alerts/test`** (POST `{method, recipient?, code?}` — tests **one method** to its current/typed recipient, no DB write), **`/api/alerts/test-alarm`** (POST `{code, methods?, rules?}` — multi-method send for one alarm using the page's *current* config snapshot, **no DB write at all**; for test/dev). Both test routes return `{ok, results}` at HTTP 200 so the page renders per-recipient outcomes. The page (`web/alerte.html`, linked in every nav as 🔔 Alertes) has a methods table (per-method recipient input + enable switch + a **Test** button enabled only when the method is active *and* its switch is on — live-wired) and an alarms switch-matrix (alarms × methods, defaults from `default_<method>`) with a per-row **Test** button (sends via the currently-checked methods without saving). `busy()` only re-enables buttons it disabled, preserving the intended-disabled test buttons. Unsaved matrix state is opt-in: **nothing is sent on real alarms until the user saves**. Note: until a running `checker_service` is restarted on this code, the new routes 404 (the static page still serves).

The dashboards (`web/statistiques.html`) have two sections: **🖥 Système** (host history chart) and **🩺 Supervision Mirth** (Mirth-process cards + process history chart — secondary sockets axis, fed by `/api/mirth/history`; an **📈 Activité Mirth** throughput chart fed by `/api/mirth/throughput` showing client-computed msg/min deltas with counter-reset clamped to 0 — with a **scope selector (`#throughput-scope`)** that retargets that single chart to *Global* / a channel / a connector (`?channel=&connector=`); and a channels table fed by `/api/mirth/api` — now **served from the latest historised snapshot** (last-known channels/connectors are shown even when Mirth is unreachable *now*, with the snapshot time displayed via `#ma-snapshot` and the status badge reflecting live reachability) — whose rows are **expandable to reveal each channel's connectors** with per-connector stats and clickable error counters that open the errors modal filtered by `channel`+`connector`). All accumulation/rendering is client-side; auto-refresh pulls only new points (`date_deb=last`). Because the heavy overview is now historised by the background collector, repeated F5 no longer fires simultaneous `get_overview()` calls at Mirth — page loads read SQLite + a tiny live `get_status`. The errors modal (`/api/mirth/messages`) and the Mirth-process cards (`/api/mirth/process`, local psutil) stay live/on-demand.

**Chart formatting is unified** across all three line charts (`buildChart`/`buildMirthChart`/`buildThroughputChart`) via shared helpers: `xTimeScale(tc)` builds an identical time x-axis whose `min`/`max` come from **`periodBounds()`** so every chart spans *exactly* the selected window (and `applyTimeBounds(ch)` re-applies it on in-place updates) — this is what makes the axis read `HH:mm` instead of `HH:mm:ss` when data is sparse. All three carry the `eventLinesPlugin` overlay (vertical event/alert bars via `chart.$eventMarkers = eventMarkers()`), including the activity chart, which re-renders on event loads via `renderThroughput()` (cached in `throughputSamples`). The period `<select id="range">` defaults to **6 h** and includes an **“Aujourd'hui”** option (`value="today"` → `currentParams()` sends `date_deb` = midnight; `periodBounds()`/`trimWindow` treat it as a non-trimmed sliding day).

**Shared theme (`web/theme.css` + `web/theme.js`).** All four pages link these instead of carrying their own `:root`/base CSS — each page keeps only its page-specific rules inline. `theme.css` is the design system: design tokens defined for both schemes (`:root` = dark default, `:root[data-theme="light"]` = light), shared components (topbar/`.brand`/nav pills, cards, panels, buttons, badges, tables, inputs, modal, console), and cross-document **View Transitions** (`@view-transition { navigation: auto; }`) for animated page-to-page navigation. **All legacy variable names are preserved** (`--cpu`, `--ok`, `--err`, `--accent`, `--panel-2`, …) so the inline `style="…var(--x)"` in each page keeps working — don't rename them. `theme.js` applies the saved/`prefers-color-scheme` theme (localStorage key `mc-theme`), wires the `.theme-toggle` button, and exposes `window.MCTheme.chartColors()` + a `mc-theme-change` event so the Chart.js axes/legend re-color on toggle (see `buildChart`/`buildMirthChart`, which read `MCTheme.chartColors()` and update colors in their in-place-update branch). Each page's `<head>` has a tiny inline anti-FOUC script that sets `data-theme` before the stylesheet loads. Chart **series** colors (`#38bdf8` etc.) stay hardcoded in JS and match the `--cpu/--mem/--disk/--sock` tokens.

## SMTP configuration & secrets (important)

`quickmail.py` resolves config in this precedence: **environment variables** (`SMTP_SERVER`, `SMTP_PORT`, `USE_SSL`, `SMTP_USER`, `SMTP_PASSWORD`, `SENDER_ADDRESS`) > `.smtp_config.py` > hardcoded defaults.

- `.smtp_config.py` holds real credentials and is **git-ignored**. `.smtp_config.py.template` is the tracked stub — copy it to bootstrap: `copy .smtp_config.py.template .smtp_config.py`.
- The config is loaded dynamically via `importlib` from `base_dir`, which is `sys._MEIPASS` when frozen by PyInstaller, else the script dir. This is what lets the compiled `.exe` find its bundled config.
- **`_compilation.bat` bakes `.smtp_config.py` into the `quickmail`, `mirth_logs_parser` and `checker_service` `.exe`s** via `--add-data ".smtp_config.py;."`. The resulting executables are self-contained and ship your real SMTP credentials inside them — do not distribute them publicly. (`system_state.exe` and `mirth_api.exe` are built without it.) Caveat: the `checker_service.exe` build bundles only the SMTP config — it does **not** bundle the `web/` static pages or `.mirth_config.py`, so the frozen service expects those next to it (the Mirth client then falls back to env vars / defaults).

`.gitignore` excludes `dist/`, `build/`, `venv/`, `__pycache__/`, `*.spec`, `.smtp_config.py`, `.mirth_config.py`, and the `checker_history.db*` files. The `.spec` files present in the tree are untracked build artifacts. The Mirth REST client follows the same secrets pattern as SMTP — see `mirth_api.py` (repo root) and `.mirth_config.py.template` in the Web service section above.
